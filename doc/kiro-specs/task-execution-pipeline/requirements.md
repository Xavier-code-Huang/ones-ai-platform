# 任务执行流水线重构 — 需求文档（v2）

> **功能名称**: task-execution-pipeline  
> **日期**: 2026-03-26  
> **版本**: v2（整合用户反馈 6 项新增需求）  
> **状态**: 待审阅

---

## 1. 问题陈述

### 背景

ones-AI 平台当前的任务执行页面存在以下核心问题：

1. **执行过程不透明**：提交任务后仅有一个命令行风格的日志流（终端黑框），用户无法直观了解"AI 现在在做什么"
2. **工单维度缺失**：多工单任务只有一个统一的日志流，无法独立跟踪每个工单的处理进度
3. **状态粒度太粗**：工单只有 pending/running/completed/failed 四个状态，缺少中间处理节点
4. **执行模式单一**：一个工单一次 claude CLI 调用，没有多 Agent 串行协作能力
5. **排队工单锁定**：多工单提交后排队中的工单无法修改，用户只能取消整个任务重建
6. **路径记忆缺失**：代码目录每次需要手动输入，不像 agent-teams 有历史记录
7. **任务提交页丑陋**：TaskView 表单设计简陋，缺少现代化交互
8. **工单卡死 Bug**：现有执行引擎在某些情况下（SSH 断连、runner 未返回 JSON）会导致后续工单永不处理

### 用户故事

- 作为**工单提交者**，我希望提交多个工单后能看到每个工单独立的处理流水线，且排队中的工单允许我修改内容
- 作为**新用户**，我希望输入工单号后 AI 能自动分析工单并生成提示词建议，我只需微调
- 作为**老用户**，我希望代码目录能记住我之前用过的路径，无需每次手动输入
- 作为**管理员**，我希望能透视任务执行的每个阶段，发现和修复卡死问题

## 2. 功能需求

### FR-100 工单独立详情页

WHEN 用户提交任务后进入任务详情页 THE SYSTEM SHALL 为每个工单展示独立的处理卡片，每个卡片包含该工单的完整处理流水线。

**验收标准**:
- [ ] 每个工单有独立的进度时间线（Timeline）
- [ ] 工单卡片按提交顺序排列
- [ ] 正在处理的工单有高亮标识和动画效果

### FR-101 处理节点时间线（参考 GitHub Actions）

WHEN 工单开始执行 THE SYSTEM SHALL 在工单详情区展示一个类似 GitHub Actions 的节点时间线，实时标记每一个处理阶段。

**设计参考**: GitHub Actions Workflow Run 视图
- 左侧 Job 列表 → 对应我们的工单列表
- 右侧 Step 时间线 → 对应每个工单的处理阶段
- 每个 Step 有状态图标（⏳🔄✅❌）、耗时、可展开详情

**预定义节点序列**:
1. 🔍 校验工单信息
2. 📁 检查代码路径
3. 📂 检查 Agent-Teams 目录
4. 🐳 启动容器环境
5. 🤖 Agent 1: 工单分析
6. 🔧 Agent 2: 代码修改
7. ✅ Agent 3: 结果验证
8. 📝 Agent 4: 报告生成与回写

**验收标准**:
- [ ] 节点有 pending/active/completed/failed/skipped 五种状态
- [ ] active 节点有脉冲动画，completed 节点有对勾标记
- [ ] 节点之间有竖向连线（已完成=实线、未完成=虚线）
- [ ] 节点可展开查看该阶段的详细输出

### FR-102 结构化执行日志

WHEN 任务执行过程中产生日志 THE SYSTEM SHALL 将日志按阶段分类展示在对应的时间线节点下，而非终端黑框。

**验收标准**:
- [ ] 日志按阶段分组，节点展开后以普通文本形式展示
- [ ] 系统消息（节点变更、超时等）以卡片/横幅形式展示
- [ ] 关键信息（AI 结论、报告摘要）突出展示
- [ ] 可通过"原始日志"按钮切换到传统终端视图

### FR-103 细粒度状态推送

WHEN 后端执行引擎处理到新阶段 THE SYSTEM SHALL 通过 WebSocket 推送 `phase_change` 事件。

**验收标准**:
- [ ] 新增 WebSocket 消息类型 `phase_change`
- [ ] 携带字段: `{ phase, status, message, timestamp, ticket_id }`
- [ ] 前端收到后实时更新对应工单的时间线节点

### FR-104 多 Agent 串行管线（Claude Subagent 架构）

WHEN 工单进入执行阶段 THE SYSTEM SHALL 支持多个 Agent 串行执行，利用 Claude CLI 的 subagent 能力实现上下文隔离和工具限制。

**Claude CLI Subagent 要点**（基于调研）:
- 每个 subagent 有独立上下文窗口，不污染主会话
- 可配置专属 system prompt 和工具白名单
- 支持串行管线：分析 → 修改 → 验证 → 回写
- 父 agent 只收到 subagent 最终结果

| 阶段 | Agent | 职责 | 输入 | 输出 |
|------|-------|------|------|------|
| 1 | 分析 Agent | 获取/分析工单 | 工单号+ONES API | 问题分析报告 |
| 2 | 修复 Agent | 修改代码 | 分析报告+代码路径 | 代码变更 |
| 3 | 验证 Agent | 检查结果 | 变更 diff | 验证结论 |
| 4 | 回写 Agent | 生成报告 | 验证结论 | 最终报告 |

**验收标准**:
- [ ] 管线阶段可通过后端配置定义
- [ ] 每个 Agent 输出作为下一个 Agent 输入
- [ ] 各 Agent 运行在同一容器内，共享文件系统
- [ ] 与用户的 agent-teams 目录不冲突（系统 agent 和用户 agent 隔离）

### FR-105 容器内部透视

WHEN 任务执行过程中 THE SYSTEM SHALL 将容器内部的执行过程以结构化方式呈现在页面上，用户感觉如同在容器内部观看。

**验收标准**:
- [ ] 不使用 `-p` 参数的批处理模式，改为流式输出解析
- [ ] 容器启动、runner 运行、claude 调用等关键事件提取并展示
- [ ] 异常信息以醒目的卡片形式展示

### FR-106 TaskDetailView 页面重构

WHEN 用户访问任务详情页 THE SYSTEM SHALL 展示全新的双栏布局：左侧工单列表+AI 结论，右侧处理流水线时间线。

**验收标准**:
- [ ] 左侧：工单列表（可切换），显示工单号、状态、AI 结论
- [ ] 右侧：选中工单的时间线流水线
- [ ] 时间线有入场动画（从上到下依次展开）
- [ ] 移动端适配为上下布局

### FR-107 代码目录历史记忆

WHEN 用户在创建任务时输入代码目录 THE SYSTEM SHALL 保存该路径到用户的历史路径列表，并在下次创建任务时提供下拉选择。

**验收标准**:
- [ ] 数据库新增 `user_code_paths` 表，存储用户历史路径
- [ ] 前端代码目录输入框支持下拉选择 + 自由输入
- [ ] 路径按使用频率排序
- [ ] 支持删除不再使用的路径

### FR-108 排队工单可编辑

WHEN 任务中有多个工单正在排队（status=pending）THE SYSTEM SHALL 允许用户修改这些排队工单的补充说明、代码目录等字段。

**验收标准**:
- [ ] 状态为 pending 的工单显示"编辑"按钮
- [ ] 可修改字段: 补充说明(note)、代码目录(code_directory)、额外挂载路径
- [ ] 正在运行(running)和已完成(completed)的工单不可编辑
- [ ] 编辑后立即保存到数据库

### FR-109 TaskView 任务提交页重构

WHEN 用户进入创建任务页面 THE SYSTEM SHALL 展示一个现代化、高度交互的任务提交界面。

**设计参考**: GitHub Actions workflow dispatch UI + 现代化表单设计

**验收标准**:
- [ ] 全新设计风格，参考 GitHub Actions 的简洁专业度
- [ ] 工单输入区支持拖拽排序
- [ ] 代码目录从历史记录下拉选择
- [ ] Agent-Teams 目录下拉选择
- [ ] 提交后平滑过渡到详情页

### FR-110 AI 工单预分析（讨论项）

WHEN 用户输入工单号后 THE SYSTEM SHALL（可选开启）调用后端 API 获取 ONES 工单标题和描述，由 AI 预分析生成提示词建议。

> [!IMPORTANT]
> **讨论项**：此功能涉及额外的 API 配额消耗和延迟，建议以"可选按钮"形式实现，而非默认自动触发。

**实现思路**:
1. 用户输入工单号 → 点击"AI 预分析"按钮
2. 后端调用 ONES API 获取工单标题+描述
3. 调用 AI（可用轻量模型）生成推荐提示词
4. 前端展示推荐提示词，用户可修改后确认

**验收标准**:
- [ ] 前端有"AI 预分析"可选按钮
- [ ] 后端 API: `POST /tasks/preview` 接收工单号列表，返回分析结果
- [ ] 分析结果包含: 工单标题、问题类型推断、建议提示词
- [ ] 用户可以直接使用或修改推荐提示词

### FR-111 工单卡死防护机制

WHEN 工单执行过程中出现 SSH 断连、runner 未返回结果等异常 THE SYSTEM SHALL 确保后续排队工单仍能正常执行。

**问题根因（排查确认）**:
- 任务 #278 中存在重复工单（同一 ticket_id 出现两次），导致打回重做逻辑紊乱
- runner 未返回 JSON 结果行时，`ticket_success` 标志未被设置，工单卡在 running
- SSH 断连后虽有重连逻辑，但重连失败时剩余工单未正确标记

**验收标准**:
- [ ] 单工单执行超时后，自动标记该工单 failed 并继续下一个
- [ ] runner 未返回 JSON 时，工单从 running 超时降级为 failed（心跳超时机制）
- [ ] 提交时去重检查，防止同一任务中出现重复工单号
- [ ] 全局定时扫描任务，清理孤立的 running 工单

## 3. 非功能需求

### NFR-100 性能
- WebSocket 推送延迟 < 500ms
- 单页面渲染 50 个工单 + 各自时间线不卡顿

### NFR-101 兼容性
- 历史任务（无 phase 数据）在新 UI 中以兼容模式展示
- WebSocket 新消息类型不影响旧客户端

### NFR-102 可维护性
- 管线阶段通过配置而非硬编码定义
- 时间线节点可通过后端配置动态扩展

### NFR-103 用户体验
- 任务提交页视觉质量参考 GitHub Actions 水准
- 表单交互流畅，无明显延迟

## 4. 范围说明

### ✅ In Scope
- 前端 TaskDetailView + TaskView 页面完整重构
- 后端 task_executor 阶段推送 + 卡死防护
- 数据库新增 `task_ticket_phases` + `user_code_paths` 表
- WebSocket 消息协议扩展
- 多 Agent 管线框架 + subagent 调研落地
- 代码目录历史记忆
- 排队工单可编辑
- AI 工单预分析（可选功能，讨论后确认）
- 工单卡死 bug 修复

### ❌ Out of Scope
- 每个专业 Agent 的具体提示词精调（后续迭代）
- 移动端原生 App
- ONES API 认证问题（上游系统）

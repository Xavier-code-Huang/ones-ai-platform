# lango-AI vs ones-AI 工作区融合分析报告

## 1. 现状：两套重复且不同步的代码

| 文件 | 工作区 A (`Ones-AI专项开发`) | 工作区 B (`lango-claude-deploy`) | 相同？ |
|------|---------------------------|-------------------------------|--------|
| `ones-AI` shell 脚本 | 243 行 | 308 行 | ❌ |
| `ones_task_runner.py` | 1278 行 | 1138 行 | ❌ |
| `Auth-Claude.sh` | ❌ 无 | 1391 行 | — |

**核心痛点**：同一个文件两个版本，没有 single source of truth，每次发版要手动对齐。

---

## 2. Auth-Claude vs ones-AI 本质差异

```
Auth-Claude.sh (1391行) — "瑞士军刀"
├── API Key 获取 ← 共享
├── Docker 容器启动 ← 共享
├── MCP 配置初始化 (Gerrit/Wiki/ONES/Android-debug)
├── Engineer DNA 规则注入
├── Hooks 热更新 + Subagents 管理
├── host-run 宿主机命令代理 (make/gcc/apt/docker...)
├── 插件同步 (marketplaces/cache)
├── tmux/Lango Remote 远程会话
└── 交互式菜单 + 首次配置向导

ones-AI.sh (243行) — "精准导弹"
├── API Key 获取 ← 共享
├── Docker 容器启动 ← 共享
├── 参数解析 (--tickets/--code-dirs/--json-output)
└── 非交互执行 (自动化场景)
```

> **结论：共享的只有「获取 Key + 启动容器」这 ~80 行胶水代码，剩余 90% 完全不同。**

---

## 3. 能否融合？

### ❌ 不建议融合，原因：

| 维度 | Auth-Claude (lango-AI) | ones-AI |
|------|----------------------|---------|
| **用户** | 全公司开发者 (~几十人) | ones-AI 平台 (自动化) |
| **使用方式** | 交互式 (人坐在终端前) | 无人值守 (SSH 远程执行) |
| **复杂度** | 极高 (MCP/Hook/DNA/代理) | 极低 (只关心执行+结果) |
| **变更频率** | 高 (每周迭代 v3.x) | 低 (稳定后很少改) |
| **对稳定性要求** | 中 (交互式出错可重启) | 极高 (后台执行不能崩) |

融合后的风险：
- Auth-Claude 的 MCP/Hook 更新可能破坏 ones-AI 的非交互模式
- ones-AI 的改动（如 `--json-output` 格式变更）可能影响全公司用户
- 1400+ 行的 shell 脚本本身就难以维护，融合只会更糟

---

## 4. ✅ 建议方案：彻底分开 + 共享基础层

### 架构

```
lango-claude-deploy/ (工作区B — lango-AI 主仓库)
├── Auth-Claude.sh          ← lango-AI 专属
├── lib/
│   └── lango-docker.sh     ← 【新】共享基础层 (Key获取+容器启动)
├── ones_task_runner/
│   └── ones-AI             ← 继续放这里但标记为"自动同步"
└── ...

Ones-AI专项开发/ (工作区A — ones-AI 平台主仓库)  
├── ones-ai-platform/       ← 平台前后端 (唯一源)
├── ones_task_runner.py      ← runner 的唯一源 (单向同步到 B)
├── ones-AI.sh               ← shell 的唯一源 (单向同步到 B)
└── ...
```

### 具体操作

#### Step 1: 确立 Single Source of Truth（唯一源）

| 文件 | 唯一源 | 方向 |
|------|--------|------|
| `ones_task_runner.py` | **工作区 A** (`Ones-AI专项开发`) | A → B 单向同步 |
| `ones-AI.sh` | **工作区 A** (`Ones-AI专项开发`) | A → B 单向同步 |
| `Auth-Claude.sh` | **工作区 B** (`lango-claude-deploy`) | 独立维护 |
| `config.yaml` | **工作区 B** (随部署包) | 独立维护 |

#### Step 2: 版本标记（解决"不知道是什么版本"）

在 `ones_task_runner.py` 和 `ones-AI.sh` 头部加版本号：

```python
# ones_task_runner.py
__version__ = "1.4.2"  # 与 ones-ai-platform 对齐
```

```bash
# ones-AI.sh
ONES_AI_VERSION="1.4.2"
```

#### Step 3: 自动同步脚本（一键发布）

创建 `sync_to_deploy.py`：
```
读取 A 的 ones_task_runner.py → 覆盖到 B/ones_task_runner/
读取 A 的 ones-AI.sh → 覆盖到 B/ones_task_runner/ones-AI
自动 git commit + push 到 B
```

#### Step 4: 部署流程简化

```
修改代码 (工作区A)
    ↓
python sync_to_deploy.py   ← 一键同步到工作区B
    ↓
在工作区B执行 pack.sh      ← 打发布包
    ↓
部署到服务器
```

---

## 5. 长期建议（可选）

如果想**彻底消除同步问题**：

- 把 `ones_task_runner.py` 和 `ones-AI.sh` 从工作区 A 移到工作区 B 的 `ones_task_runner/` 下
- 工作区 A (ones-ai-platform) 只包含平台代码（前端+后端）
- `task_executor.py` 通过 SSH 调用远程服务器的 `ones-AI`，无需本地有 runner 副本
- 这样每个仓库职责单一：A=平台，B=CLI工具集

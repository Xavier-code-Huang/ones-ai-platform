### 1. 容器名捕获正则**过于宽泛** — 会误匹配

[task_executor.py:375](file:///f:/Ones-AI专项开发/ones-ai-platform/backend/task_executor.py#L375)

```python
if not container_name and ("ones-ai-" in line):
    _m = _re.search(r'(ones-ai-[\w-]+)', line)
```

> [!CAUTION]
> 改成 `"ones-ai-" in line` 后，**任意包含 `ones-ai-` 字样的日志行都会触发捕获**。
> 
> 比如 runner 执行过程中如果 Claude 输出类似 `"建议参考 ones-ai-lichao 的配置"` 之类的日志，也会被误匹配。
> 
> 虽然有 `if not container_name` 保护（只捕获一次），但**第一次触发可能不是容器名行**。特别是 `task_executor.py:287` 的 `_save_log` 会输出执行命令：
> ```
> 执行命令: export LANG=C.UTF-8 ... ones-AI --json-output ...
> ```
> 这行里可能包含 `ones-ai-` 但不是容器名！

**修复方案**: 匹配关键字应该更精确：

```python
# 匹配 runner 实际输出的两种格式
if not container_name and ("容器:" in line or "容器: " in line or "Entering container" in line):
```

或者用正则直接匹配完整的容器名格式 `ones-ai-用户名-PID`：

```python
if not container_name:
    _m = _re.search(r'\bones-ai-\w+-\d+\b', line)
    if _m:
        container_name = _m.group(0)
```

这里 `\d+` 约束了末尾必须是 PID 数字，可以排除大部分误匹配。

## 🟡 中等问题 (建议修复)

### 3. 观测模式 `docker logs -f` 执行完后前端终端不会自动关闭

[terminal_ws.py:126](file:///f:/Ones-AI专项开发/ones-ai-platform/backend/terminal_ws.py#L126)

```python
cmd = f"docker logs -f --tail 100 {container_name}"
```

> [!IMPORTANT]
> 当任务容器执行完毕后（容器停止），`docker logs -f` 进程会自然退出（EOF）。此时 SSH 进程结束，WebSocket 的 `_ssh_to_ws` 循环也会退出。
> 
> 但 `_ws_to_ssh` 方向（用户终端输入 → SSH stdin）**在观测模式下完全无用**，用户键盘输入会被发到 `docker logs -f` 的 stdin —— 这个进程根本不读 stdin。虽然不会出错，但白白消耗资源。
> 
> 更关键的是：如果容器还没启动（刚创建但没运行），`docker logs -f` 可能直接返回空然后退出，用户看到的就是终端一闪而过。

**建议**: 观测模式下禁止用户输入方向的转发，并在终端显示一个提示文案。

# 综合修复：模型选择 + 观测容器 + 孤儿任务自动重试 + SSH

## 诊断结果总结

### 问题1: 模型选择（使用 Opus）

`ones-AI.sh` 第 228-231 行，runner 启动 docker 时设置了环境变量：
```bash
-e "ANTHROPIC_BASE_URL=https://open.bigmodel.cn/api/anthropic"
-e "ANTHROPIC_DEFAULT_HAIKU_MODEL=glm-4.7"
-e "ANTHROPIC_DEFAULT_SONNET_MODEL=glm-4.7"
-e "ANTHROPIC_DEFAULT_OPUS_MODEL=glm-5"
```

Claude Code 通过 `ANTHROPIC_DEFAULT_SONNET_MODEL` 选择**默认**使用的模型。当前默认是 `glm-4.7`（对应 Sonnet 级别）。要让 claude 默认使用 Opus 模型，有两种方案：

- **方案A（推荐）**: 在 runner 的 `claude -p` 调用中加 `--model` 参数，显式指定使用 opus。需要修改 `ones_task_runner` 的 runner 代码（不在平台控制范围内）。
- **方案B**: 把 `ANTHROPIC_DEFAULT_SONNET_MODEL` 也改成 `glm-5`，这样所有默认调用都走 Opus。但这可能导致成本增加。

> [!IMPORTANT]
> 需要你确认：是**所有任务都用 Opus**，还是**在平台上提供一个模型选择下拉框**让用户按任务选择？
> 如果是前者，直接改 runner 环境变量；如果是后者，需要在前端新建任务时增加模型选择 UI。

### 问题2: 观测容器看不到运行日志

**根因：容器名捕获正则不匹配！**

`task_executor.py` 第 367-368 行：
```python
if not container_name and ("正在进入容器" in line or "Entering container" in line):
```

但 `ones-AI.sh` 实际输出的是（第 209 行）：
```
[*] 启动 ones-AI (容器: ones-ai-lichao-211778)...
```

关键字不匹配！`"正在进入容器"` 和 `"启动 ones-AI (容器:"` **完全不同**。

结果就是：后端永远捕获不到容器名 → `container_name` 永远为空 → 前端按钮永远停在"⏳ 容器准备中" → 用户只能在任务结束后点"干预容器"进去，而**观测容器按钮永远不可用**。

由于观测模式下 `terminal_ws.py` 用的是 `docker logs -f`，这个本身是正确的，问题只在于容器名捕获不到。

### 问题3: 孤儿任务应自动重试

当前重启恢复逻辑（已修改）是：把 running 工单标 failed，让用户手动打回重做。

用户要求改为：**自动将被中断的 running 工单重置为 pending，让它们重新排队执行**。这样后端重启后，被中断的任务会自动恢复执行，用户无需任何操作。

同时，任务级状态不应该标记为 `completed`，应保持 `running` 以确保 executor 继续执行。

### 问题4: SSH 失败原因

**你之前在脚本和我的诊断脚本中用的密码 `lc` 是错误的。**

后端实际使用的 SSH 凭据存在 `user_server_credentials` 表中，字段 `ssh_password_encrypted` 是经过 AES 加密的。后端通过 `crypto.decrypt_password()` 解密后连接。

你给的 `lg` 用户 + `5;bspQ5rGk+4R(EA` 密码在编译机 192.168.1.29 上是有效的，但从你的 Windows 机器直连 192.168.1.29 不通（网络隔离），需要通过 172.60.1.35 跳板。

诊断脚本中的 SSH 失败不是程序 bug，**后端平台本身的 SSH 连接是正常工作的**（后端日志可以看到 "Auth for user lichao succeeded"）。

### 🔴 BUG-1: Agent 路径含中文时 glob 可能失败

[ones_task_runner.py:341](file:///F:/Claude/lango-claude-deploy/ones_task_runner/ones_task_runner.py#L341)

```python
for f in sorted(glob.glob(pattern)):
```

在某些 Linux 环境下，`glob.glob()` 默认不匹配中文路径。Python 3.12 已修复，但 Docker 容器中的 Python 版本可能低于此。

**风险等级**: 中（取决于 Docker 镜像的 Python 版本）
**建议**: Agent 目录名一般不含中文，但 MD 文件名可能含中文。保险起见加个 fallback：

```python
# 如果 glob 没结果，尝试 os.walk
if not prompts and os.path.isdir(agent_dir):
    for root, dirs, files in os.walk(agent_dir):
        depth = root.replace(agent_dir, '').count(os.sep)
        if depth > 1: continue  # 只扫一级子目录
        for name in sorted(files):
            if name.endswith('.md'): ...
```

### 🔴 BUG-2: `shlex.quote` 对含单引号的 agent_dir 路径在嵌套 shell 中可能出错

[task_executor.py:117](file:///F:/Ones-AI专项开发/ones-ai-platform/backend/task_executor.py#L117)

```python
cmd = " ".join(shlex.quote(p) for p in cmd_parts)
```

SSH 执行的命令被 paramiko 通过 `exec_command` 发送，paramiko 内部会将整个命令字符串传给远程 shell。`shlex.quote` 生成 `'...'` 包裹的字符串。如果路径本身含单引号（如 `/home/user/it's-a-dir`），`shlex.quote` 会正确转义为 `'it'"'"'s-a-dir'`，但这在**二次嵌套**的 ones-AI 脚本中（`build_args → ARGS_FILE → mapfile`）能否正确还原存疑。

**风险等级**: 低（路径含单引号的场景极罕见）
**建议**: 暂不处理，记录为已知限制。

## 第 2 轮：设计冲突和一致性检查

### ✅ 无冲突项

- kiro spec `task-detail-enhance` 改的是 `task_executor.py` 的**JSON 解析**部分（L227+）和 `TaskDetailView.vue`，不涉及命令构建和路径处理 → **确认无冲突**
- `agent_dir` 存在 `tasks` 表（任务级），`extra_mounts` 存在 `task_tickets` 表（工单级）→ 粒度合理

### 🟡 DESIGN-1: `extra_mounts` 前后端存储不一致

| 层 | 格式 |
|----|------|
| 前端 | `string[]` 数组 |
| 后端 API 接收 | `list[str]` |
| 数据库存储 | 逗号分隔字符串 `TEXT` |
| task_executor 读取 | `t.get("extra_mounts").split(',')` |
| ones-AI 接收 | `--extra-mounts "a,b,c"` 逗号分隔 |

**问题**: 如果路径本身含逗号（虽然 Linux 路径允许含逗号但极罕见），整条链路会错误拆分。

**风险等级**: 低
**建议**: 在 `validate_path()` 中额外检查路径是否含逗号，若有则拒绝。

### 🟡 DESIGN-2: `agent_dir` 是任务级，`extra_mounts` 是工单级

`agent_dir` 对整个任务只有一个值，但 `extra_mounts` 每个工单可能不同。

`task_executor.py:195-201` 中将所有工单的 `extra_mounts` **合并去重**后一起传给 `ones-AI`：

```python
all_extra_mounts = set()
for t in tickets:
    mounts_str = t.get("extra_mounts") or ""
    if mounts_str:
        for m in mounts_str.split(','):
            if m.strip():
                all_extra_mounts.add(m.strip())
```

这意味着即使某工单没用到某路径，该路径也会被挂载到 Docker 中。这是**可接受的设计权衡**（Docker 挂载是容器级的，不能按工单拆分），记录为已知行为。

## 第 3 轮：代码问题和隐性 Bug

### 🔴 BUG-3: `load_agent_prompts` 文件句柄泄漏

[ones_task_runner.py:347](file:///F:/Claude/lango-claude-deploy/ones_task_runner/ones_task_runner.py#L347)

```python
content = open(f, encoding='utf-8').read().strip()
```

`open()` 没有 `with` 或 `close()`。虽然 CPython 的引用计数会立即关闭，但 PyPy 或其他实现不会。如果 Agent 目录有很多 MD 文件（>100），可能耗尽文件描述符。

**修复**:
```python
with open(f, encoding='utf-8') as fh:
    content = fh.read().strip()
```

### 🔴 BUG-4: `extra_mounts` 空字符串在数据库中和空列表在逻辑上的混淆

**写入路径** (tasks.py:151):
```python
extra_mounts_str = ','.join(m for m in t.extra_mounts if m)
```
如果 `extra_mounts = []`，`extra_mounts_str = ""`（空字符串）。

**读取路径** (task_executor.py:197-201):
```python
mounts_str = t.get("extra_mounts") or ""
if mounts_str:
    for m in mounts_str.split(','):
```

`"".split(',')` 返回 `['']`（**一个空字符串元素的列表**，不是空列表！）。但因为有 `if m.strip():` 过滤，所以不会出问题。

**结论**: 逻辑上安全，但容易让后续开发者误解。建议加注释。

### 🟡 HIDDEN-1: `ones-AI` 脚本中 `ARGS_ARRAY` 解析位置和 `build_args` 解析位置的差异

ones-AI 脚本中有两处参数解析：
- L169 `ARGS_ARRAY=("$@")` — 原始参数（在 Docker 挂载前）
- L237 `build_args "$@"` — 处理后参数（写入 args 文件传给容器）

我们在 L167-178 用 `ARGS_ARRAY` 解析 `--agent-dir` 和 `--extra-mounts`。这是**正确的**，因为这些参数是给 Docker 挂载用的（容器外），不是给容器内用的。

但存在一个微妙问题：`build_args` 也会把 `--agent-dir` 和 `--extra-mounts` 传给容器内的 runner。runner 的 argparse 也认识这两个参数。这意味着**同一个参数既在脚本外用于挂载，又在脚本内用于提示词加载**。这是**设计正确的**——两端各取所需。

### 🟡 HIDDEN-2: Agent 目录在 HOME/WORK_DIR 下时不额外挂载但 runner 仍能访问

```bash
case "$AGENT_DIR" in
    "${USER_HOME}"*|"${WORK_DIR}"*) ;;  # 已被 HOME 挂载覆盖，跳过
    *) MOUNT_ARGS+=(-v "${AGENT_DIR}:${AGENT_DIR}:ro") ;;
esac
```

当 Agent 目录在 HOME 下时，跳过额外挂载但 HOME 已经被 `-v ${USER_HOME}:${USER_HOME}` 挂载了，所以容器内可以访问。**但此时 Agent 目录不是只读的**（HOME 挂载是 rw）。如果要求 Agent 目录必须只读，需要修改逻辑。

**风险等级**: 低（只读是安全加固，非必须）

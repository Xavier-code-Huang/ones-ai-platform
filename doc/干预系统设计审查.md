# 干预系统设计审查

## 当前存在的问题

### 场景分析

用户点"唤醒干预"时，存在 **3 种容器状态**：

| 状态 | 容器名举例 | 含义 |
|------|-----------|------|
| 任务容器 **运行中** | `ones-ai-zengying-12345` | ones-AI 正在执行任务 |
| 任务容器 **已停止** | `ones-ai-zengying-12345` | 任务已完成，容器退出 |
| 干预容器 **已停止** | `ones-ai-zengying-intervene-67890` | 上次干预后退出 |

### 问题 1: tasks.py — 对已停止的任务容器执行 `docker start` 是错误的

当前代码（第 754-758 行）：
```python
# 容器存在但已停止，启动它
res = await ssh_conn.run(f"docker start {container_name}")
```

任务容器的主进程是 `ones_task_runner.py → claude -p "..."`。
`docker start` 会重新执行这个主进程，等于**重新跑一次任务**，不是干预。

**正确做法**：区分干预容器和任务容器：
- 干预容器（名含 `intervene`）→ 直接 `docker start`（主进程是 bash，安全）
- 任务容器（已停止）→ 不要 start，创建新的干预容器

### 问题 2: terminal_ws.py — 运行中的任务容器用 `docker attach` 无法干预

当前代码（第 127-128 行）：
```python
else:
    cmd = f"docker attach --sig-proxy=false {container_name}"
```

`docker attach` 连接到主进程的 stdin/stdout。主进程是非交互的 `claude -p`，
用户只能看输出，**不能输入、不能操作文件、不能杀进程**。

**正确做法**：对运行中的任务容器也用 `docker exec -it bash`，给用户一个独立 shell。
用户可以在里面查看文件、启动自己的 claude、或 `kill` 正在运行的 claude 进程。

## 修正后的设计

### tasks.py `start_ticket_container`

```
用户点 "唤醒干预"
  ↓
查 DB 获取 container_name
  ↓
是干预容器？（名含 intervene）
  ├── YES → docker inspect
  │    ├── running → 返回 "已在运行"
  │    ├── stopped → docker start → 返回 "已唤醒"
  │    └── 不存在 → 创建新干预容器
  └── NO（任务容器）
       ├── running → 返回 "任务正在执行中"（前端直接开终端）
       └── stopped/不存在 → 创建新干预容器
```

### terminal_ws.py 连接逻辑

```
WebSocket 连接
  ↓
查 DB 获取 container_name + conversation_id
  ↓
容器名含 intervene？
  ├── YES（干预容器）
  │    ├── 有 conversation_id → claude --resume --conversation-id <id>
  │    └── 无 conversation_id → bash --rcfile auto_resume.sh（首次）
  └── NO（任务容器，正在运行中）
       → docker exec -it bash（独立 shell，不干扰主进程）
```

### 断开时的 conversation_id 捕获

仅对干预容器执行（任务容器不需要）。
